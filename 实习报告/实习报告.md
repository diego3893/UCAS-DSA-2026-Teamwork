## 2.4 马踏棋盘
### 1. 需求分析
- **输入的形式和输入值的范围**：
读者可自行指定一个马的初始位置 $(i, j)$，坐标范围为 $0 \le i, j \le 7$。
- **输出的形式**：
求出马的行走路线后，将数字 1, 2, …, 64 依次填入一个 8×8 的方阵并输出；在可视化模式下，支持演示寻找行走路线的回溯过程。
- **程序所能达到的功能**：
将马随机放在 8×8 棋盘的某个方格中，按走棋规则移动，每个方格只进入一次，走遍全部 64 个方格。程序需采用**非递归**实现。此外支持选作功能：求出多条以至全部行走路线；探讨每次选择位置的“最佳策略”以减少回溯次数。
- **测试数据**：
由读者指定，可自行指定任意合法的 $(i, j)$ 进行测试。

### 2. 概要设计
本程序采用非递归的深度优先搜索 (DFS) 结合 Warnsdorff 启发式策略。下面列出程序中使用的抽象数据类型定义、主程序流程以及各程序模块间的层次与调用关系：

#### 2.1 抽象数据类型定义
程序中主要定义了用于模拟系统栈的栈节点结构体 `StackNode`，其结构如下：
```c
typedef struct {
    int x, y;   // 马的当前坐标
    int dir;    // 马下一步的方向索引 (在排序后的候选列表中的位置)
} StackNode;
```
基于此结构体，程序手动维护了一个容量为 64 的显式栈用于控制 DFS 的前进与回溯状态：
```c
static StackNode stack[64]; // 棋盘最多走64步
static int top = -1;
```

主程序 `main` 的整体控制流如下：

3. **运行参数配置**：提示用户选择 4 种运行模式之一（单条路径、多条路径、过程可视化等），并根据用户的输入设定目标求解数量 `maxSolutions` 和是否开启可视化的标识 `verbose`。

程序采用模块化设计，以 `main` 函数为入口，核心逻辑集中在 `solve` 函数中。整体调用层次关系如下：

### 3. 详细设计

#### 3.1 候选选择策略（Warnsdorff 启发式）
为减少无效搜索，程序在每一步展开前，会计算所有合法下一步候选格子的“出度”（即下一步还能走的其他合法格子数量），并将候选格子按照启发式规则进行排序。排序的主键是“出路的度数越小越好”，次键是“距离棋盘中心越远越好”，这极大降低了陷入死局的概率。核心代码实现如下：

```c
// 收集所有合法的下一步候选格子
for (int i = 0; i < 8; i++) {
    int nx = cur->x + dx[i];
    int ny = cur->y + dy[i];
    if (nx >= 0 && nx < 8 && ny >= 0 && ny < 8 && board[nx][ny] == 0) {
        int deg = countDegree(nx, ny);
        candidates[count].x = nx;
        candidates[count].y = ny;
        candidates[count].deg = deg;
        count++;
    }
}

// Warnsdorff 启发式排序：
// 优先选择度数小（出路少）的格子；度数相同时，优先选择距离中心较远的格子
for (int i = 0; i < count - 1; i++) {
    for (int j = i + 1; j < count; j++) {
        int d1 = centerDistance(candidates[i].x, candidates[i].y);
        int d2 = centerDistance(candidates[j].x, candidates[j].y);
        if (candidates[j].deg < candidates[i].deg || 
           (candidates[j].deg == candidates[i].deg && d2 > d1)) {
            // 交换位置
            Candidate temp = candidates[i];
            candidates[i] = candidates[j];
            candidates[j] = temp;
        }
    }
}
```

#### 3.2 死路预判（剪枝）
在遍历排序好的候选格子前，程序会进行死路预判。如果某个候选格子的出度为 0，并且当前搜索尚未达到 63 步（即并非跳向最后一步终点），说明走入该格后将陷入死胡同，此时程序直接将其剪枝跳过：

```c
// 遍历排序后的候选列表
for (int i = cur->dir + 1; i < count; i++) {
    int nx = candidates[i].x;
    int ny = candidates[i].y;
    int deg = candidates[i].deg;

    // 死路剪枝：如果无路可走且还没到倒数第一步，必然是死局，直接跳过
    if (deg == 0 && step < 63) continue;
    
    // ... 执行进栈和状态转移逻辑
}
```

#### 3.3 回溯机制的核心控制
程序通过手动维护栈 `stack` 来代替系统的函数调用栈。栈中每个节点不仅保存了马的坐标 `(x, y)`，还记录了已经尝试过的方向索引 `dir`。若在某一层中所有的候选方向都尝试失败（或被剪枝），程序将执行恢复现场、步数递减和弹栈的操作，回退到上一状态继续探索：

```c
int moved = 0;
// 接着上次尝试的方向继续向后遍历
for (int i = cur->dir + 1; i < count; i++) {
    // ... [进行前述的剪枝操作] ...
    
    cur->dir = i; // 记录当前栈节点已尝试到第 i 个候选
    push((StackNode){nx, ny, -1}); // 将新的位置作为新节点入栈
    board[nx][ny] = ++step;        // 更新棋盘记录并增加步数
    moved = 1;                     // 标记已成功找到下一步
    break; // 深入下一层，中止当前节点的水平扩展
}

// 回溯处理：若当前节点无法向任何方向移动
if (!moved) {
    board[cur->x][cur->y] = 0; // 撤销当前位置的棋盘标记
    pop();                     // 弹出当前节点
    step--;                    // 步数回退
}
```

#### 3.4 多路线与可视化支持
当步数 `step` 累加至 64 时，说明已成功找到一条遍历整个棋盘的解。为了支持多路径模式（或穷举全部解），程序在输出当前解后，会主动模拟一次“回溯”（即将栈顶节点弹出，撤销最后一步），以迫使算法继续寻找其他可能的分支。同时程序加入了键盘侦听，以便在海量搜索时能够被用户随时安全中断：

```c
// 检查是否走满 64 步，找到一条完整路径
if (step == 64) {
    solutionsFound++;
    printBoard(); // 输出当前解

    // 如果运行模式设定为只寻找一条路径，直接返回成功状态
    if (maxSolutions == 1) return 1; 

    // 检测用户是否按下了 'Q' 键中止程序
    if (checkAbort()) { userAbort = 1; break; }

    // 寻找多条路径的核心策略：主动回退一步，继续深搜下一个分支
    board[stack[top].x][stack[top].y] = 0;
    pop();
    step--;
    continue;
}
```


### 4. 调试分析

#### 4.1 算法复杂度分析
- **时间复杂度**：国际象棋 8×8 棋盘的骑士周游路线总数超过 260 亿条。如果不加任何优化地进行暴力深度优先搜索，最坏情况下的时间复杂度高达 $O(8^{64})$，枚举整个状态空间是不现实的。但是，在本程序中引入 Warnsdorff 启发式策略后，对 8×8 棋盘极为有效。实测表明，几乎所有的起点都能在**零回溯或极少量回溯**的情况下瞬间找到第一条完整路径。
- **空间复杂度**：由于程序按要求采用了**非递归**的实现方式，避免了系统调用栈的深度递归带来的栈溢出风险。程序仅显式地分配了一个容量为 64 的 `StackNode` 数组作为辅助栈，因此空间复杂度为严格的 $O(64)$（常数级）。
  
  ```c
  // 空间复杂度 O(64) 的代码体现
  static StackNode stack[64]; // 棋盘最多走64步
  static int top = -1;
  ```

#### 4.2 启发式策略（Warnsdorff规则）的效能分析
在调试初期若仅使用传统的按固定顺序（如顺时针探索 8 个方向）的回溯法，程序很容易在最后几步陷入死局，产生海量的回溯。通过引入排序机制，算法性能得到了质的飞跃：
1. **主键优化（出路的度数）**：优先走“出路最少”的格子。这种贪心策略直观上保证了棋盘边缘和角落的格子能够被尽早访问，防止后期这些位置成为无法到达的死胡同。
2. **次键优化（中心距离度量）**：当多个候选格子的出路度数相同时，程序通过计算放大的曼哈顿距离 `abs(2 * x - 7) + abs(2 * y - 7)`，优先让马走向距离棋盘中心更远的位置。这一策略进一步降低了算法在棋盘中央陷入死胡同的概率。

#### 4.3 剪枝与状态监控分析
在寻找多条路径（或不限数量模式）的调试过程中，由于解的空间极为庞大，程序很容易陷入长时间的搜索。对此，本程序做了以下调试和控制设计：
- **死路预判剪枝**：如果一个未走完 63 步的候选位置出度为 0（即 `deg == 0 && step < 63`），程序会将其判定为死路并直接剪枝，跳过入栈操作。这有效削减了无效的搜索分支。
- **安全中断机制**：在死循环或海量搜索时，程序利用 `GetAsyncKeyState('Q')` 实现了非阻塞的键盘侦听。在调试和实际运行中，用户按下 `Q` 键即可优雅退出并输出当前的耗时、节点和回溯统计信息，避免了强制结束进程导致的数据丢失。
  ```c
  // 检查用户是否按 Q 中止
  if (checkAbort()) { userAbort = 1; break; }
  ```
- **耗时精确统计**：程序在搜索入口处利用 `<time.h>` 库的 `clock()` 函数记录 `t0` 和 `t1`，从而精确输出搜索耗费的 CPU 时间，为评估算法效能提供了直观的数据支撑。

### 2. 概要设计

#### 2.1 抽象数据类型定义
为了避免繁琐的指针操作和动态内存碎片，程序采用数组（`std::vector`）加静态内存池的方式来存储树节点。定义了两种树节点结构：

```cpp
// 1. AVL 树节点结构 (严格平衡)
struct TreeNode{
    int key;    // 键值
    int height; // 绝对高度
    int bf;     // 平衡因子 = 右子树高度 - 左子树高度
    int left;   // 左孩子下标
    int right;  // 右孩子下标
};

// 2. FHQ Treap 节点结构 (期望平衡，用于选作)
struct TreapNode {
    int key;      // 键值
    int priority; // 随机优先级 (维护堆性质)
    int size;     // 子树大小
    int left;
    int right;
};
```

为实现节点的循环利用，程序维护了一个删除节点回收站。每次新建节点优先从回收站取用废弃下标，避免无限扩容。

3. **接受指令**：提供包括插入、删除、查找、基于 Treap 的分裂合并演示、基于 AVL 的分裂合并演示以及退出等操作菜单。
4. **功能分发**：根据用户选择，读取相应参数并调用 `tree/AVL.cpp` 或 `tree/treap.cpp` 中的核心算法，反馈操作结果。

### 3. 详细设计

#### 3.1 AVL树的插入与旋转调整

在执行普通的二叉搜索树插入后，程序向上回溯更新节点的高度（`height`）和平衡因子（`bf = 右树高 - 左树高`）。若判定 `bf < -1`，说明左子树过高，再根据左孩子的平衡因子进一步判断是单次右旋还是先左旋后右旋（LR旋转）；右侧过高同理。

```C++
    updateNode(idx); // 更新插入后的节点信息

    if(tree[idx].bf < -1){ // 左侧更高
        if(tree[tree[idx].left].bf <= 0){
            return rightRotate(idx); // 单次右旋处理 LL 型不平衡
        }else{
            int newLeft = leftRotate(tree[idx].left); // LR 型不平衡
            tree[idx].left = newLeft;
            return rightRotate(idx);
        }
    }
    // ... 对称处理右侧更高的 LL 和 RL 旋转 ...
```

#### 3.2 AVL树的删除与替代机制

删除算法采用了类似 B 树的替代删除策略。如果待删节点只有一棵子树或为叶子节点，则直接将其删除并由子节点顶替。如果待删节点有两棵子树，程序会进入右子树一直向左找，找到它的**中序直接后继**。将后继的键值覆盖到当前节点，随后递归地在右子树中删除该后继节点，最后向上回溯更新并旋转：

```C++
        }else{ // 如果两棵子树
            // 找它的中序直接后继（右子树的最左节点）
            int successor = tree[idx].right;
            while(tree[successor].left){
                successor = tree[successor].left;
            }
            tree[idx].key = tree[successor].key; // 将后继复制过来
            
            // 递归删除右子树的后继节点
            int nextRight = removeNode(tree[idx].right, tree[successor].key);
            tree[idx].right = nextRight;
        }
```

#### 3.3 选作：树的分裂与合并的两种实现方式

本程序对选作内容“树的分裂与合并”给出了两套截然不同的实现方案：

##### 方案一：基于 AVL 树的严格平衡重构

**设计思路**：遍历源树，将节点逐个拆下，并使用 `insertNode` 插入到目标树中（分裂时根据值域判断插入哪棵新树）。

```C++
// AVL 分裂：前序遍历原树，将节点根据 val 判断插入到左树或右树中
static void splitTraversal(int srcNode, int val, int& outLeft, int& outRight){
    if(!srcNode) return;
    int leftChild = tree[srcNode].left;
    int rightChild = tree[srcNode].right;
    bool dummy = false;
    
    if(tree[srcNode].key <= val)
        outLeft = insertNode(outLeft, tree[srcNode].key, dummy);
    else
        outRight = insertNode(outRight, tree[srcNode].key, dummy);
        
    deleteNode(srcNode);
    splitTraversal(leftChild, val, outLeft, outRight);
    splitTraversal(rightChild, val, outLeft, outRight);
}

// AVL 合并：遍历将一棵树的节点逐一插入到另一棵树中
static int mergeTraversal(int destRoot, int srcNode){
    if(!srcNode) return destRoot;
    int leftChild = tree[srcNode].left;
    int rightChild = tree[srcNode].right;
    bool dummy = false;
    
    destRoot = insertNode(destRoot, tree[srcNode].key, dummy);
    deleteNode(srcNode);
    destRoot = mergeTraversal(destRoot, leftChild);
    destRoot = mergeTraversal(destRoot, rightChild);
    return destRoot;
}
```

##### 方案二：基于 FHQ Treap 的期望平衡拆接

**设计思路**：无旋 Treap 本身就是由 `Split` 和 `Merge` 两个基础操作构建的。分裂时，根据二叉搜索树的性质直接断开并重接指针；合并时，根据随机生成的 `priority` 维护大根堆性质重接两棵树，期间不涉及任何旋转操作。

```C++
// Treap 分裂：直接根据权值打断并重连子树指针
pair<int, int> FHQ_split(int root, int val){
    if(!root) return {0, 0};
    if(fhq_tree[root].key <= val){
        pair<int, int> p = FHQ_split(fhq_tree[root].right, val); 
        fhq_tree[root].right = p.first; // 将<=val的部分接在第一棵树上
        updateTreapNode(root);
        return {root, p.second};
    }else{
        pair<int, int> p = FHQ_split(fhq_tree[root].left, val); 
        fhq_tree[root].left = p.second;
        updateTreapNode(root);
        return {p.first, root};
    }
}

// Treap 合并：根据 priority 维护堆性质重连
int FHQ_merge(int x, int y){
    if(!x || !y) return x ? x : y;
    if(fhq_tree[x].priority > fhq_tree[y].priority){ 
        fhq_tree[x].right = FHQ_merge(fhq_tree[x].right, y); 
        updateTreapNode(x);
        return x;
    }else{
        fhq_tree[y].left = FHQ_merge(x, fhq_tree[y].left); 
        updateTreapNode(y);
        return y;
    }
}
```

### 4. 调试分析

#### 4.1 复杂度分析

- **时间复杂度**：AVL 树的基础操作（查找、插入、删除）通过严密的平衡因子维护和旋转操作，保证了在最坏情况下严格的 $O(\log n)$ 性能。
- **空间复杂度**：采用 `std::vector` 实现的静态内存池，节点数为 $n$ 时占用 $O(n)$ 空间，不会引发由于频繁 `new/delete` 导致的内存碎片。

#### 4.2 AVL 与 Treap 分裂合并实现的优劣对比

本程序针对分裂和合并功能完整实现了 AVL 与 Treap 两种方案，通过对比调试得出以下分析：

- **AVL 树方案**：
  - **优势**：分裂或合并后，结果必定是一棵极其严格的平衡二叉树，保证了后续所有搜索、插入等操作的时间下限最优。代码复用率高，只需重复调用 `insertNode` 即可完成。
  - **劣势**：耗时巨大。将树 A 拆碎后逐一插入树 B 中，时间复杂度达到了 $O(m \log(n+m))$，在面对海量数据的大树分裂和合并时性能表现吃力。
- **FHQ Treap（无旋树堆）方案**：
  - **优势**：专为区间和集合的批量操作而生。由于只需要顺着路径重定向指针，不涉及任何单节点的重复开辟与插入，无论是 Split 还是 Merge 都直接拥有期望 $O(\log n)$ 的极高效率。
  - **劣势**：不再刻意维持左右子树高度差绝对 $\le 1$ 的物理平衡，而是通过 `rand()` 优先级维护堆性质来达到“期望平衡”。在极其罕见的极端随机数下，树可能退化，得不到绝对性能保障。
